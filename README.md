# 🎧 Multi-Agent Customer Support System

A production-ready multi-agent AI system built for a customer support use case, combining **LangGraph**, **Model Context Protocol (MCP)**, **RAG (Retrieval-Augmented Generation)**, and **Groq** to intelligently route and answer questions about customer data and company policies.

---

## 📌 Project Overview

This system demonstrates a **supervisor-worker multi-agent architecture** where:

- A **Supervisor** (LangGraph) analyzes each user question and routes it to the right specialist agent
- A **SQL Agent** handles structured queries about customers, tickets, and orders from a SQLite database
- A **RAG Agent** answers policy questions by searching semantically over indexed company PDF documents
- An **MCP Server** (Model Context Protocol) exposes all tools to both agents via a standardized stdio interface
- A **Streamlit UI** provides a clean chat interface for end users

The system can handle compound questions that require both agents — for example, *"What's the return policy, and does customer #5 have any open tickets?"* — by invoking both specialists in sequence and combining their answers.

## 🏗 Architecture

```
User (Streamlit UI)
        │
        ▼
  Supervisor Node  ◄──────────────────────┐
  (LangGraph)                             │
        │                                 │
   Routing logic                          │
   (keyword + LLM fallback)               │
        │                                 │
   ┌────┴─────┐                           │
   ▼          ▼                           │
SQL Agent   RAG Agent                     │
(ReAct)     (ReAct)                       │
   │          │                           │
   └────┬─────┘                           │
        │                          (loop back to supervisor)
        ▼
  MCP Server (stdio)
  ┌──────────────────────────────────────┐
  │  query_customer_profile              │
  │  query_tickets                       │
  │  query_orders                        │
  │  run_sql                             │
  │  search_policy_documents             │
  └──────────────────────────────────────┘
        │                   │
        ▼                   ▼
  SQLite DB           ChromaDB
  (customers,         (PDF embeddings)
   tickets, orders)
```

## 🛠 Tech Stack

| Layer | Technology |
|---|---|
| LLM | [Groq](https://groq.com/) — `openai/gpt-oss-120b` (see note below) |
| Agent orchestration | [LangGraph](https://github.com/langchain-ai/langgraph) |
| Agent framework | LangChain ReAct (`create_react_agent`) |
| Tool protocol | [Model Context Protocol (MCP)](https://modelcontextprotocol.io/) via `langchain-mcp-adapters` |
| Vector store | [ChromaDB](https://www.trychroma.com/) |
| Embeddings | `sentence-transformers/all-MiniLM-L6-v2` (HuggingFace, local) |
| Database | SQLite |
| PDF loading | `pdfplumber` via `langchain-community` |
| UI | [Streamlit](https://streamlit.io/) |

## ⚙️ Setup Instructions

### 1. Prerequisites

- Python 3.10+
- A [Groq API key](https://console.groq.com/) (free tier available)

### 2. Clone the Repository

```bash
git clone https://github.com/YOUR_USERNAME/YOUR_REPO_NAME.git
cd YOUR_REPO_NAME
```

### 3. Create and Activate a Virtual Environment

```bash
python -m venv venv

# macOS / Linux
source venv/bin/activate

# Windows
venv\Scripts\activate
```

### 4. Install Dependencies

```bash
pip install -r requirements.txt
```

### 5. Configure Environment Variables

Create a `.env` file in the project root:

```env
GROQ_API_KEY=your_groq_api_key_here
```

### 6. Seed the Database

This creates `data/customers.db` with 200 fake customers, 800 tickets, and 500 orders:

```bash
python src/setup_db.py
```

### 7. Add Policy PDFs

Place your company policy PDF documents in `data/policies/`. The system was built and tested with Apple return, refund, warranty, and AppleCare+ policy documents.

### 8. Ingest PDFs into the Vector Store

This chunks all PDFs, creates embeddings, and writes to `data/chroma_db/`:

```bash
python src/ingest.py
```

> **Note:** The first run downloads the `all-MiniLM-L6-v2` embedding model (~90 MB). Subsequent runs are fast.

- *"What's the refund policy for defective items, and does customer #1 have any open tickets about defective products?"*

## 🧩 How the Routing Works

The `supervisor_node` uses a **two-layer routing strategy**:

1. **Keyword matching** — fast, deterministic detection of policy vs. SQL intent using curated keyword lists
2. **LLM fallback** — for genuinely ambiguous questions, the supervisor calls the Groq LLM with a structured output schema (`RoutingDecision`) to decide

The supervisor tracks which agents have already responded via the `called` list in `AgentState`, preventing redundant calls and enabling compound question handling.

## ⚠️ Known Limitations & Notes

- **Groq model name:** This project uses `openai/gpt-oss-120b`, which is an OpenAI-compatible model served through Groq's API. The `openai/` prefix is part of Groq's routing syntax, not a reference to OpenAI directly. If this model is unavailable on your Groq account, swap it for another supported model (e.g. `llama3-70b-8192`) in the `llm = ChatGroq(model=...)` line in `supervisor.py`, `sql_agent.py`, and `rag_agent.py`.
- **Rate limiting:** Groq's free tier has per-minute token limits. The supervisor includes exponential back-off retry logic (`_retry_on_ratelimit`), but compound questions that hit both agents may occasionally slow down on the free tier.
- **Read-only SQL:** The `run_sql` MCP tool enforces SELECT-only queries. INSERT/UPDATE/DELETE/DROP are blocked.
- **Local embeddings:** Embeddings run entirely on CPU using HuggingFace. No external embedding API is required.
- **PDF quality:** RAG quality depends on the quality and structure of the policy PDFs. Scanned/image-based PDFs will not extract well without OCR preprocessing.
